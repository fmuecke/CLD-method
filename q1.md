> Q1: Research the most common pitfalls and false assumptions that can happen during software development, starting with assuming the wrong problem, jumping to the wrong solution, testing the wrong things, or testing at the wrong level, iterating too less. So my idea is to, for the journey of software development, for every step that we need to take, according to, for example, the V model, define one common fallacy that agile or iterating and feedback loops used to close.

A1:

Your framing is directionally right, but slightly idealized: Agile doesn’t “solve” these fallacies—it just shortens the feedback loop so you notice them earlier. Many teams still fail in exactly the same ways, just faster.

Below is a structured view along a simplified V-model (left = definition, right = validation), with one dominant fallacy per stage and what iterative approaches try to counteract.

---

# 1. Problem Understanding (Requirements / Intent)

**Typical fallacy:**
*“We understand the problem because stakeholders said so.”*

**What actually goes wrong:**

* Stakeholders describe solutions, not problems
* Needs are inferred, not observed
* Edge cases and context are missing

**Agile countermeasure:**

* Continuous discovery (interviews, observation, usage data)
* Problem framing as hypotheses, not facts
* Early validation with real users

**Residual risk:**
Teams still accept backlog items as “truth” without validation.

---

# 2. Solution Framing (System / Architecture / Design)

**Typical fallacy:**
*“We can design the right solution upfront.”*

**What actually goes wrong:**

* Over-engineering or wrong abstractions
* Optimizing for imagined scale or constraints
* Lock-in through early architectural decisions

**Agile countermeasure:**

* Incremental architecture (“evolve, don’t predict”)
* Spikes and prototypes
* Deferring irreversible decisions

**Residual risk:**
“Emergent design” becomes accidental design without intentional guidance.

---

# 3. Specification (Detailing / Backlog / Stories)

**Typical fallacy:**
*“If it’s written down, it’s understood.”*

**What actually goes wrong:**

* Ambiguity in acceptance criteria
* Missing non-functional requirements
* Misaligned mental models between roles

**Agile countermeasure:**

* Collaborative refinement (PO + Dev + QA)
* Example-based specification (ATDD)
* Thin slices instead of big specs

**Residual risk:**
Teams treat stories as mini-specs instead of conversation starters.

---

# 4. Implementation (Coding)

**Typical fallacy:**
*“If it compiles, it works.”*

**What actually goes wrong:**

* Local correctness vs system correctness
* Hidden coupling and side effects
* Technical debt introduced unnoticed

**Agile countermeasure:**

* TDD / pair programming
* Continuous integration
* Small batch sizes

**Residual risk:**
TDD becomes test-after instead of test-first; CI becomes just a build server.

---

# 5. Unit Testing (Left-side verification)

**Typical fallacy:**
*“High coverage means high confidence.”*

**What actually goes wrong:**

* Testing implementation details instead of behavior
* Mock-heavy tests that don’t reflect reality
* False sense of safety

**Agile countermeasure:**

* Behavior-focused tests
* Testing critical paths, not everything
* Refactoring with safety nets

**Residual risk:**
Teams optimize for coverage metrics instead of meaningful tests.

---

# 6. Integration Testing

**Typical fallacy:**
*“Components that work individually will work together.”*

**What actually goes wrong:**

* Contract mismatches
* Environment differences
* Timing / concurrency issues

**Agile countermeasure:**

* Early and frequent integration
* Contract testing
* Shared environments / pipelines

**Residual risk:**
Integration is still postponed due to environment complexity.

---

# 7. System Testing (End-to-End)

**Typical fallacy:**
*“End-to-end tests validate the system sufficiently.”*

**What actually goes wrong:**

* E2E tests are brittle and slow
* Limited coverage of real scenarios
* Hard to diagnose failures

**Agile countermeasure:**

* Test pyramid (focus lower levels)
* Selective E2E for critical journeys
* Observability instead of only testing

**Residual risk:**
Teams either over-rely on E2E or avoid it completely.

---

# 8. User Validation (Acceptance / Release)

**Typical fallacy:**
*“If it meets requirements, users will adopt it.”*

**What actually goes wrong:**

* Feature ≠ value
* Usability gaps
* Misaligned incentives or workflows

**Agile countermeasure:**

* Early releases / beta users
* Usage analytics
* Feedback loops (not just sign-off)

**Residual risk:**
“User acceptance” is reduced to PO acceptance.

---

# 9. Iteration / Learning Loop

**Typical fallacy:**
*“We are iterating, therefore we are learning.”*

**What actually goes wrong:**

* Iterations without real feedback
* Output-focused sprints (velocity theater)
* No explicit hypothesis → no learning

**Agile countermeasure:**

* Inspect & adapt (retrospectives, metrics)
* Hypothesis-driven development
* Short feedback cycles

**Residual risk:**
Iterations become mechanical cycles without actual adaptation.

---

# Synthesis (what your idea boils down to)

Each stage has a **systematic cognitive bias**:

| Stage           | Core bias                   |
| --------------- | --------------------------- |
| Problem         | Assumed understanding       |
| Solution        | Premature certainty         |
| Specification   | Illusion of clarity         |
| Implementation  | Local optimization          |
| Unit Test       | Metric illusion             |
| Integration     | Composition optimism        |
| System Test     | Coverage illusion           |
| User Validation | Requirement = value fallacy |
| Iteration       | Activity = learning fallacy |

---

# Critical observation

Your model implicitly assumes a sequence (like V-model), but most of these fallacies happen **continuously and in parallel**.

A more accurate framing would be:

> Software development is a continuous process of invalidating assumptions at different levels.

Agile doesn’t eliminate fallacies—it just:

* reduces batch size
* increases feedback frequency
* lowers cost of being wrong
