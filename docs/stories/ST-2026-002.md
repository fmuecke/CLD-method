# ST-2026-002: Verify BM↔ADO Epic linking round-trip

Initiative: INIT-2026-002
Hypothesis: HYP-2026-001

## Slice

A verification and bootstrap script scoped to a specific BM board that:
1. Queries all cards on the BM board and all Epics in the ADO work area
2. Computes the delta: which ADO Epics are missing from BM (→ create), which BM cards have no CustomId (→ log gap), which are paired
3. Creates missing BM cards for unmatched ADO Epics (bootstrap) — write path proven by card creation
4. Verifies paired cards by fetching the ADO work item and confirming it exists — read path proven

BM is source of truth for Epic content (title, description). The sync never overwrites BM card data from ADO.

Produces a report: total ADO Epics, total BM cards, newly created, already paired, unlinked BM cards (no CustomId).

## Expected Behavior

- Given a BM board ID and an ADO work area,
  When the script runs,
  Then it reports: total ADO Epics, total BM cards, newly created in BM, already paired, BM cards with no ExternalId.

- Given an ADO Epic with no matching BM card (ExternalId not found on board),
  When the script runs,
  Then a new BM card is created on the target board with ExternalId set to the ADO Epic ID and title copied from ADO.

- Given an ADO Epic that already has a matching BM card,
  When the script runs,
  Then the ADO work item is fetched to confirm it exists, the card is counted as paired — BM card data is not modified.

- Given a BM card on the board with no ExternalId,
  When the script runs,
  Then it is logged as an unlinked card — not touched, not deleted, explicit report entry.

- Given a BM card whose ExternalId cannot be fetched from ADO,
  When the script runs,
  Then the failure is logged with the card details — not swallowed silently.

## Smallest Possible Test

- **Test type:** Integration — runs against live BM and ADO instances. No meaningful unit test is possible for a linking verification.
- **Why this level is sufficient:** The claim is about the integration path working end-to-end. A mock would only prove that the script runs, not that the systems are actually connected. The write test uses the ADO title as the payload — no synthetic or test-only data needed.

## Decision Type

- [x] **Fast-lane** — diagnostic script; the title write is the only mutation and reflects real data (ADO title → BM title). Easily reversible.

## Scope Boundary

**In scope:**
- Query all cards on a specific BM board (`BM_BOARD_ID`)
- Query all Epics in a specific ADO work area (`ADO_AREA_PATH` or project)
- Compute delta: paired / ADO-only (missing in BM) / BM-only (no ExternalId)
- Create BM cards for ADO-only Epics (bootstrap); write ExternalId and title
- Write ADO title to existing paired BM cards (proves write path)
- Console report: total ADO / total BM / created / paired / unlinked

**Out of scope:**
- Syncing any field other than title — state/column sync is ST-2026-001
- Polling or scheduling — this is a one-shot bootstrap/verification script
- Deleting BM cards — the sync never removes cards
- State category mapping — that is ST-2026-001

## Constraints

- BM credentials: API token via environment variable (`BM_API_TOKEN`, `BM_BOARD_ID`)
- ADO credentials: PAT via environment variable (`ADO_PAT`, `ADO_ORG`, `ADO_PROJECT`, `ADO_AREA_PATH`)
- No credentials in source code or output
- Script is idempotent: running it twice does not create duplicate BM cards
