# ST-2026-001: Compute BusinessMap Epic state from ADO child state categories

Initiative: INIT-2026-002
Hypothesis: HYP-2026-002

## Slice

Given a BusinessMap Epic with a valid ExternalId (ADO Epic ID), read the state categories of all ADO child items and compute the correct BusinessMap Epic state using the agreed mapping rules. No polling loop, no BM write — just the mapping function with a verifiable output.

## Expected Behavior

**Mapping rules (ADO state category → BusinessMap column):**
- Proposed → "Requested"
- InProgress → "In Progress"
- Complete → "Done"
- Removed → excluded from calculation

**State derivation:**
- Given an Epic whose ADO children are all in category Proposed,
  When the mapping function runs,
  Then the computed BM column is "Requested".

- Given an Epic whose ADO children include at least one InProgress (and any mix of Proposed, Complete, Removed),
  When the mapping function runs,
  Then the computed BM column is "In Progress".

- Given an Epic whose ADO children are all Complete (Removed excluded),
  When the mapping function runs,
  Then the computed BM column is "Done".

- Given an Epic whose ADO children are all Removed,
  When the mapping function runs,
  Then the Epic is skipped — no BM state change — and logged for manual review. (BM card removal/archiving is a separate mechanism, not a column state; deferred to a later story.)

- Given an Epic whose ADO children are a mix of Complete and Proposed (none InProgress),
  When the mapping function runs,
  Then the computed BM column is "In Progress" (partial completion = still in progress).

**Open constraint:** The sync must not move an Epic *backward* (e.g. from "In Progress" to "Requested"). Sync only advances state or holds it. Backlog is out of scope for the sync — it is the pre-planning state before an Epic is linked to ADO.

## Smallest Possible Test

- **Test type:** Unit tests against the mapping function. Integration test against a real ADO project with known child states to verify the state category field is returned as expected.
- **Why this level is sufficient:** The mapping logic is pure — given a list of state categories, produce an Epic state. Unit tests cover all five scenarios above deterministically. The integration test confirms the ADO API actually returns the `stateCategory` field in the expected format.

## Decision Type

- [x] **Fast-lane** — the mapping function is a pure computation with no external side effects. Easy to change if rules evolve.

## Scope Boundary

**In scope:**
- Reading ADO child item state categories via the ADO REST API for a given Epic ID
- The mapping function: list of state categories → Epic state (or "flag")
- Unit tests for all five scenarios
- One integration test against a real ADO project

**Out of scope:**
- Writing the computed state back to BusinessMap — next story
- Polling loop or scheduling — later
- Handling Epics with no children — deferred (define behavior when needed)
- BusinessMap API calls of any kind

## Constraints

- Must query the `System.State` and its category via the ADO REST API — do not infer category from state name strings, as that would break with custom states.
- The "all Removed" flag must be a distinct output, not silently treated as any of the three main states.
