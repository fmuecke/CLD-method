# INIT-2026-002: BusinessMap ↔ ADO Epic Sync

Status: **Discovery** | Delivery | Validating | Done

## Problem

<!-- What user or business problem are we addressing? Describe the problem, not a solution.
     If you catch yourself writing "build X" or "add Y" — stop. That's a solution.
     Rephrase as the pain or gap that X or Y would address. -->

_What is broken, missing, or worth improving? Be specific about the symptom, not the solution._

BusinessMap is being introduced as the company-wide tool for strategic items (Initiatives, Epics, OKRs). Azure DevOps (ADO) remains the system of record for development execution (Features, PBIs, Tasks). These two systems have no automated connection.

When they diverge, company alignment breaks down: development has its own truth and works on whatever seems fit. WIP limits cannot be enforced at the value-stream level. Flow is impeded and transparency is undermined — strategic priorities and actual development work become decoupled.

## Who Experiences It

_Which users, roles, or systems are affected? How often? At what scale?_

At least all the people working within the value stream and need to coordinate around those items:

- Product Management (PM)
- Product Marketing Management (PMM)
- Technical Product Owners (PO)
- Commercial Operations (Com-Ops)
- Consulting/Presales
- Sales
- Management

Frequency: That can be multiple times a week or daily.

## Evidence It Exists

_What data, observation, or signal confirms this is real — not hypothetical? "We think" is not evidence._

Departmental misalignment and local optimization were severe enough to justify adopting a value stream approach and a shared board. The physical board already demonstrated that shared visibility improves coordination and focus — teams are more "ready to sell" when delivery happens. The risk is regression: without a sync between BusinessMap and ADO, the two systems diverge and the alignment gains are lost.

## Target Outcome

_What observable change would indicate success? How would we measure it?_

- BusinessMap is the source of truth for Epic management. No manual updates to Epic column/state required.
- ADO Epics without a BM counterpart are automatically created in BM on first sync (initial bootstrap only). After that, BM owns the Epic lifecycle — the sync never auto-deletes BM cards.
- When work starts on ADO child items (Features, PBIs, Tasks), the parent Epic in BusinessMap moves to "in progress" within minutes to 2 hours — without manual intervention.
- WIP limits in BusinessMap reflect actual development activity, enabling flow control across the value stream.
- ~20 Epics/week sync without errors or manual correction.

## Non-goals

_What are we explicitly not solving? This prevents scope creep._

- Managing ADO child items (Features, PBIs, Tasks) in BusinessMap — ADO remains the system of record for execution detail.
- Syncing Initiatives between both tools — BusinessMap Initiatives stay in BusinessMap only.
- Deleting BM Epics automatically — the sync never removes cards from BM. Removal is a deliberate human decision.
- Improving the value stream process itself.

## Open Assumptions

_List what you are taking for granted. These become the seeds for HYP-\*.md documents._

**Confirmed (de-risked):**
- BusinessMap REST API supports read/write on Epic state — already tested
- ADO REST API supports read/write on work items — already tested
- BusinessMap also exposes an MCP server for board item access

**Still assumed (candidates for HYP):**
- ADO child item states map to BusinessMap Epic states via these rules (defined, not yet validated against actual API state names):
  1. Any child "in progress" → Epic "in progress"
  2. All children "done" → Epic "done"
  3. Some children "done", others not started → Epic "in progress"
  4. No child worked on → Epic not "in progress"
  - Residual assumption: ADO work item states map cleanly onto three categories (not started / in progress / done) without ambiguous intermediate states
- BM's ExternalId/CustomId field stores the ADO Epic ID — this is the stable link between both systems
- **Epic creation direction:** BM is the initiating system for new Epics going forward. However, the sync is also responsible for the initial bootstrap: ADO Epics without a BM counterpart are created in BM on first run, with the ADO ID written to ExternalId.
- After bootstrap, the sync never auto-creates or auto-deletes BM Epics — only column/state is updated.
- Residual assumption: the manual linking workflow (BM → create ADO Epic → write ADO ID back to BM) is low-friction enough that it won't be skipped in practice
- A polling-based sync (minutes to 2h latency) is sufficient; webhooks or real-time push are not required for Phase 1
- HYP-2026-001: BM Epics can be matched to ADO Epics via ExternalId and all linked Epics are present in both systems
- HYP-2026-002: ADO work item states map cleanly to three sync categories (Confirmed)
- HYP-2026-003: Polling at ≤2h latency is sufficient for WIP control
- HYP-2026-004: The manual Epic linking workflow is reliable enough in practice
