# HYP-2026-002: ADO work item states map cleanly to three sync categories

Status: Confirmed
Initiative: INIT-2026-002

## Belief

We believe that every ADO work item state can be unambiguously classified into one of three sync categories — **not started**, **in progress**, **done** — by querying the ADO state category field rather than individual state names, making the mapping robust to custom states.

## Why We Think This

ADO's state category system maps every state — including custom ones — to one of three built-in categories: **Proposed**, **InProgress**, **Complete**. Querying this field rather than state names means the mapping is defined by ADO, not maintained by us. Confirmed from ADO documentation.

## What Changes If We're Right

The state-mapping logic can be implemented as a simple lookup table with no branching for ambiguous states. No manual override mechanism is needed. The sync can be fully automated.

## Falsification Signal

We find an ADO work item state — actually used in a project feeding into the sync — that does not fit any of the three categories, or that produces a wrong Epic state in BusinessMap when the mapping rule is applied. For example: a "Blocked" or "In Review" state that is neither clearly "in progress" nor "done".

## Why a Positive Result Is Not Trivial

Even if all current states map cleanly, teams can add custom states at any time. A clean mapping today is not a guarantee of a clean mapping next quarter. Confirmation only covers the current state vocabulary — the sync still needs a strategy for unknown states introduced later.

## Alternative Hypothesis

ADO projects have enough state customization that at least one project feeding into the sync uses a state that doesn't fit the three categories. The mapping requires a configurable fallback or a per-project override, not a single lookup table.

## Smallest Possible Test

- **Method:** Query the ADO REST API for all distinct work item states across every project that will feed into the sync. Enumerate them and manually classify each into not started / in progress / done. Flag any that are ambiguous.
- **Confirms if:** Every state in use classifies unambiguously into one of the three categories with no disagreement between the people doing the classification.
- **Rejects if:** One or more states are ambiguous or unclassifiable without additional context or a judgment call.

## Time to Evidence

Hours — one API call per project, manual review of the state list. Can be done before writing any sync logic.

## Confidence Before Testing

Medium — the default ADO state model is simple, but custom states are common enough that we shouldn't assume clean mapping without checking.

## Validation Plan

1. Enumerate all states via API (Smallest Possible Test above).
2. If any ambiguous states exist: decide on a classification rule or fallback behavior, and add that decision to the sync specification.
3. After the sync is running: monitor for unclassified states hitting the fallback — signals that new custom states were introduced.

## Stories Derived From This Hypothesis

- ST-2026-001: Compute BusinessMap Epic state from ADO child state categories

## Result

- Status: Confirmed
- What was observed: ADO does not use substates — it uses **state categories**. Every ADO state, including custom ones, maps to one of three categories: **Proposed** (→ not started), **InProgress** (→ in progress), **Complete** (→ done). The sync should query the state category field, not state names. This makes the mapping robust by design — custom states are handled automatically by ADO's own categorization.
  - **Removed** is a known additional category outside the three main ones. Decision: exclude Removed items from Epic state calculation — if scope was removed, it does not influence the state of remaining work. Edge case: an Epic where *all* children are Removed should be flagged for manual review rather than marked "done".
- Decision: Confirmed. Implement sync against ADO state categories. Full mapping:
  - Proposed → BM "Requested"
  - InProgress → BM "In Progress"
  - Complete → BM "Done"
  - Removed (all children) → skip Epic, log for manual review. BM card removal is a separate mechanism (not a column state); deferred.
  - Sync never moves an Epic backward; Backlog is pre-planning only.
