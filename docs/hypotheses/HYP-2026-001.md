# HYP-2026-001: BM Epics can be matched to ADO Epics via ExternalId and all linked Epics are present in both systems

Status: Open
Initiative: INIT-2026-002

## Belief

We believe that:
1. BusinessMap Epics with a populated ExternalId can be reliably matched to their ADO counterpart by ID.
2. ADO Epics without a BM counterpart can be detected by comparing the ADO Epic set against the BM board's ExternalId set, and created in BM as part of an initial bootstrap.
3. A basic read-write round-trip (syncing the ADO title into BM) works correctly for all linked Epics.

## Why We Think This

Both APIs have been tested individually. The ExternalId field exists in BusinessMap and is intended for exactly this kind of cross-system reference. However, the full round-trip — query BM, find ExternalId, fetch ADO work item, verify match — has not been run end-to-end. This is an assumption about integration, not individual APIs.

## What Changes If We're Right

The linking mechanism is proven and the sync foundation is solid. All subsequent stories (state sync, polling loop) can be built on top with confidence that the fundamental join between the two systems works.

## Falsification Signal

One or more of:
- A BM Epic has an ExternalId set but the corresponding ADO work item cannot be fetched (wrong ID format, wrong project, deleted item)
- A BM Epic has no ExternalId — missing from the sync without any error
- Title written to ADO does not appear in BM (or vice versa) — the write path is broken
- The set of linked BM Epics does not match expectations (Epics present in one system but not the other)

## Why a Positive Result Is Not Trivial

Even if the round-trip works for the Epics tested, there may be Epics with malformed or missing ExternalIds that silently fall out of the sync. Confirmation requires checking completeness — not just that *some* Epics sync, but that *all expected* Epics are accounted for.

## Alternative Hypothesis

The ExternalId field is populated inconsistently — some Epics were created before the linking workflow was established, or the workflow was skipped under time pressure. The sync works for well-linked Epics but silently misses a meaningful fraction, creating a false sense of completeness.

## Smallest Possible Test

- **Method:** Query BM for all Epics. For each, check ExternalId is set. For each linked Epic, fetch the ADO work item by ID and confirm it exists and is of type Epic. Then write the ADO title back to the BM Epic's description (or a test field) and verify it appears. Report: total Epics, linked count, unlinked count, failed fetches.
- **Confirms if:** All expected Epics have a valid ExternalId, all ADO fetches succeed, and the test write round-trips correctly.
- **Rejects if:** Any Epic is missing an ExternalId, any ADO fetch fails, or the write does not appear in BM.

## Time to Evidence

Hours — a short script against the live BM and ADO instances. No production changes needed if using a read-only test or a scratch field for the write test.

## Confidence Before Testing

Medium — the individual APIs are proven but the integration path has not been walked end-to-end.

## Validation Plan

1. Run the Smallest Possible Test against real BM and ADO instances.
2. If any Epics are unlinked: decide whether to fix the data, enforce linking going forward, or add a sync report for unlinked Epics (feeds into HYP-2026-004).
3. If all linked and round-trip works: close as Confirmed, proceed to ST for state sync.
