> Q10: Let's make CLD better!

A10:
Good. Then don’t iterate on CLD as-is. Tighten the weak spots.

Below is a **targeted hardening**, not a rewrite.

---

# 1. Fix the biggest flaw: “evidence ≠ truth”

## Problem

Your system validates:

- that _a test exists_

But not:

- that the test is **diagnostic**

## Upgrade: “Discriminating Evidence”

Change the core rule:

> Not “smallest possible test”
> but **“cheapest test that could realistically falsify the hypothesis”**

Force this explicitly in `HYP-*`:

```md
## Falsification criteria

What result would make this hypothesis unlikely or wrong?

## Why this test is discriminating

Explain why a positive result is not trivial or misleading.
```

### Effect

- Kills “click = value” type fake validation
- Forces thinking about **false positives**

---

# 2. Add a competing hypothesis (mandatory)

## Problem

Single hypotheses drift into confirmation bias.

## Upgrade: “At least one alternative”

Every `HYP-*` must include:

```md
## Alternative hypothesis

What else could explain the problem?

## Why we prefer the current hypothesis

What evidence makes this more plausible?
```

### Effect

- Forces divergence before convergence
- Prevents solution-locking

---

# 3. Move from “traceability” → “decision audit”

## Problem

You track artifacts, not decisions.

## Upgrade: Decision-centric model

Introduce a first-class object:

```
DEC-*
```

Every PR must answer:

```md
## Decision

What are we choosing to believe now?

## Evidence used

Links to EVID-\*

## Confidence level

Low / Medium / High

## Reversal trigger

What would make us revisit this?
```

### Effect

- Makes uncertainty explicit
- Enables later correction (critical for learning loops)

---

# 4. Add a “post-evidence sanity check”

## Problem

Evidence is accepted too easily.

## Upgrade: Review must challenge evidence

Extend `review.agent.md`:

Instead of:

> “does the test validate the hypothesis?”

Add:

- Could this result happen if the hypothesis is wrong?
- What’s the most likely alternative explanation?
- What signal is still missing?

### Effect

- Introduces adversarial thinking
- Reduces false confidence

---

# 5. Split hypothesis types (you’re mixing them)

Right now everything is a “hypothesis”. That’s too coarse.

## Upgrade: classify hypotheses

Add type:

```md
## Type

- Behavioral (user behavior)
- Value (user benefit)
- Technical (system behavior)
- Operational (performance, reliability)
```

### Why this matters

Each type needs different evidence:

| Type        | Typical valid evidence |
| ----------- | ---------------------- |
| Behavioral  | observation, usage     |
| Value       | outcome, retention     |
| Technical   | tests                  |
| Operational | metrics                |

### Effect

Prevents:
→ validating a **value hypothesis with a unit test**

---

# 6. Introduce “irreversible vs reversible” gating

## Problem

You apply the same rigor everywhere.

## Upgrade: decision class

In `ST-*`:

```md
## Decision type

- Reversible (low cost to undo)
- Irreversible (high cost / user impact)
```

### Rules

- Reversible → lightweight CLD (fast lane)
- Irreversible → full CLD (strict evidence)

### Effect

- Reduces friction
- Focuses rigor where it matters

---

# 7. Add a “time-to-evidence” constraint

## Problem

CLD can drift into analysis-heavy upfront work.

## Upgrade:

```md
## Time to evidence

How quickly can we get signal? (target: hours/days/weeks)
```

Hard rule:

> Prefer worse test sooner over perfect test later

### Effect

- Keeps loop tight
- Prevents over-design

---

# 8. Detect gaming (especially by AI)

## Problem

Agents will generate compliant nonsense.

## Upgrade: introduce heuristics in review

Flag if:

- hypothesis == implementation phrasing
- test only validates happy path
- no plausible falsification defined
- alternative hypothesis is weak/strawman

You can codify some of this in CI:

- length / similarity checks
- missing sections
- trivial phrasing patterns

But most:
→ belongs in review agent behavior

---

# 9. Allow “exploration mode” (currently missing)

## Problem

CLD assumes you already understand enough to form hypotheses.

That’s often false.

## Upgrade:

Introduce explicit mode:

```
EXP-*
```

Rules:

- no hypothesis required
- goal = generate hypotheses
- outputs:
  - observations
  - candidate hypotheses

### Effect

- avoids fake precision early
- keeps CLD honest

---

# 10. Align with trunk-based development

## Problem

You risk reintroducing phases.

## Upgrade: make CLD incremental

Rule:

> Each commit should reduce uncertainty — not complete a phase

So instead of:

```
HYP → ST → IMPL → EVID
```

Allow:

```
commit = micro-hypothesis + micro-test + micro-evidence
```

### Effect

- CLD becomes continuous, not staged
- aligns with trunk-based flow

---

# What CLD becomes after hardening

Not:

> a gated workflow

But:

> a **decision system with embedded skepticism**

---

# Minimal delta (if you only change 3 things)

If you want leverage, do only this:

1. Add **falsification criteria**
2. Require **alternative hypothesis**
3. Introduce **decision + confidence + reversal trigger**

That alone fixes ~70% of the weaknesses.
