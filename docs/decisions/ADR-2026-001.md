# ADR-2026-001: BusinessMap ↔ ADO Epic Sync Flow

Status: **Accepted**
Initiative: INIT-2026-002

## Context

BusinessMap (BM) is the company-wide tool for strategic items. Azure DevOps (ADO) is the system of record for development execution. A sync process keeps them consistent so that WIP limits in BM reflect actual development activity.

BM is the source of truth for Epic metadata (title, description, etc.).
ADO is the source of truth for execution state (child item state categories).

## Sync Flow

The sync runs on a polling interval (target: minutes to ≤2h — see HYP-2026-003). Each run is idempotent: running it twice produces the same result and does not create duplicate items. Errors are per-item: a failing Epic is skipped and logged; the run continues and will retry on the next poll cycle.

The flow has two sequential phases: **reconcile first, then sync state**. Reconciliation ensures a clean set of linked Epics before state sync runs.

> **Bootstrap vs. ongoing sync:** The initial one-time bootstrap (ST-2026-002) creates BM cards for ADO-only Epics. The ongoing sync described here assumes bootstrap has completed — group c behavior differs between the two (see below).

---

### Phase 1 — Reconcile Epic existence

#### 1.1 Fetch

- Read all Epic cards from the target BM board (`BM_BOARD_ID`)
- Read all non-Removed Epics from the target ADO area path (`ADO_AREA_PATH`, UNDER semantics — includes child paths)

#### 1.2 Classify

Partition into four groups based on BM `custom_id` ↔ ADO Epic ID matching:

| Group | Condition |
|---|---|
| **a) Linked** | BM card has `custom_id` AND matching ADO Epic exists |
| **b) BM-only** | BM card has no `custom_id` set |
| **c) ADO-only** | No BM card has a `custom_id` matching this ADO Epic ID |
| **d) Stale link** | BM card has `custom_id` but no ADO Epic with that ID exists |

#### 1.3 Resolve groups b, c, d

**Group b — BM-only (no ADO counterpart):**
- Create an ADO Epic with title from BM
- Write the new ADO Epic ID back to the BM card's `custom_id`
- Card moves into group a for the remainder of this run

**Group c — ADO-only (no BM counterpart):**
BM is source of truth. After bootstrap, an ADO-only Epic means the BM card was intentionally deleted.
- Set the ADO Epic state to Removed
- Log the action

> Bootstrap exception: `Invoke-BmAdoBootstrap.ps1` (ST-2026-002) creates BM cards for ADO-only Epics on initial sync. The ongoing sync never creates BM cards.

**Group d — Stale link (BM card has `custom_id` but ADO Epic not in query):**
The ADO Epic was either set to Removed or truly deleted. BM card still exists — BM is authoritative.
- Fetch the ADO Epic directly by ID (regardless of state):
  - **Found, state = Removed:** restore it — update from BM (title), set state back to active
  - **Not found (truly deleted):** create a new ADO Epic from BM data (title); update BM card's `custom_id` with the new ID
- Log the action in both cases

---

### Phase 2 — Sync state for linked Epics (group a)

Runs against all linked Epics — those already linked at fetch time plus any promoted from groups b and c in Phase 1.

#### 2.1 Metadata sync (BM → ADO)

BM is the source of truth. Write BM field values to the ADO Epic on every run:
- Title (BM → ADO)

Conflict rule: **BM wins.** Edits made directly in ADO are overwritten on next sync.

#### 2.2 State sync (ADO → BM column)

Read child work items (Features, PBIs, Tasks) of the ADO Epic. Compute the target BM column from ADO state categories (HYP-2026-002, ST-2026-001):

| ADO child state categories | BM column |
|---|---|
| All Proposed | Requested |
| Any InProgress | In Progress |
| All Complete (Removed excluded) | Done |
| Mix of Complete + Proposed | In Progress |
| All Removed | Skip — flag for manual review |

Update the BM card column only if it differs from the current value. **Never move a card backward** — column order is fixed: Backlog → Requested → In Progress → Done → Ready to Archive.

---

### Phase 3 — Report

Emit a structured log after each run:
- Counts per group (linked / BM-only / ADO-only / stale)
- Items created in BM / ADO
- Column updates applied
- Items skipped due to errors (with reason)
- Stale links flagged

## Error Handling

- Per-item errors: caught, logged, skipped — sync continues
- A run with partial failures reports as "completed with errors" — failed items retry on next poll cycle
- Auth failures: abort immediately, no partial work attempted

## Invariants

- The sync **never deletes** items from either system
- The sync **never moves a BM card backward** in column order
- **BM wins** on metadata conflicts — ADO edits are overwritten
- Running the sync twice produces the same result (idempotent)

## Field Mapping Reference

| Operation | Fields |
|---|---|
| BM → ADO (ongoing, existing) | Title only |
| ADO → BM (bootstrap / group d restore, new card) | Title + Description |
| ADO → BM (ongoing, existing card) | State/column only — never overwrites BM metadata |
